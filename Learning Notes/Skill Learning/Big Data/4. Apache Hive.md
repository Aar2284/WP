## 1. What is Apache Hive?

**Apache Hive** is a distributed, fault-tolerant data warehouse software system built on top of Apache Hadoop. It facilitates reading, writing, and managing large datasets residing in HDFS using SQL-like syntax called **HiveQL (Hive Query Language)**. 

### Why Hive?
MapReduce requires developers to write low-level, complex Java code. Hive abstracts this complexity by automatically translating SQL queries into directed acyclic graphs (DAGs) of execution jobs (using MapReduce, Apache Tez, or Apache Spark engines).

```
   [User CLI / JDBC Client]
             |
             v (HiveQL Query)
     ==================
     |  Hive Server2  |
     ==================
       /              \
      v                v (Fetch Schema)
[Query Compiler]   [Hive Metastore] (MySQL/PostgreSQL RDBMS)
      |
      v (Translate to execution plan)
[Execution Engine] ---> (Launches YARN Containers / MapReduce jobs)
      |
      v (Reads/Writes raw data)
   [HDFS]
```

### The Hive Metastore
Unlike traditional databases that package storage and metadata together, Hive decouples them:
* **Data Storage**: Stored as raw flat files in HDFS (typically in `/user/hive/warehouse`).
* **Metadata Storage**: Table definitions, schemas, columns, partitioning paths, and statistics are stored in an external relational database (like MySQL, PostgreSQL, or Derby) known as the **Hive Metastore**.

## 2. Hive Data Types

Hive supports a wide array of data types, classified into **Primitive** and **Complex** types.

### A. Primitive Data Types

| Category | Type | Size / Range | Description |
| :--- | :--- | :--- | :--- |
| **Numeric** | `TINYINT` | 1-byte integer | Range: -128 to 127 |
| | `SMALLINT` | 2-byte integer | Range: -32,768 to 32,767 |
| | `INT` | 4-byte integer | Range: -2B to 2B |
| | `BIGINT` | 8-byte integer | Range: -9B to 9B |
| | `FLOAT` | 4-byte single-precision | Floating-point value |
| | `DOUBLE` | 8-byte double-precision | Floating-point value |
| | `DECIMAL` | Fixed-point decimal | Syntax: `DECIMAL(precision, scale)` |
| **String** | `STRING` | Variable-length | Text string |
| | `VARCHAR` | Variable-length | Syntax: `VARCHAR(length)` (Max 65355 characters) |
| | `CHAR` | Fixed-length | Syntax: `CHAR(length)` (Padded with spaces) |
| **Date/Time** | `TIMESTAMP` | YYYY-MM-DD HH:MM:SS.fffffffff | Nanosecond precision timestamp |
| | `DATE` | YYYY-MM-DD | Date format |
| **Miscellaneous**| `BOOLEAN` | `TRUE` / `FALSE` | Logical state |
| | `BINARY` | Byte array | Raw binary data (images/files) |

### B. Complex / Collection Data Types

| Complex Type | Syntax & Definition | Real-world Example Query Usage |
| :--- | :--- | :--- |
| **`ARRAY`** | An indexable ordered collection of fields of the same type. <br>`ARRAY<data_type>` | `locations ARRAY<STRING>` <br>Accessed via index: `locations[0]` |
| **`MAP`** | An unordered collection of key-value pairs. Keys must be primitives. <br>`MAP<primitive_type, data_type>` | `salaries MAP<STRING, FLOAT>` <br>Accessed via key: `salaries['manager']` |
| **`STRUCT`** | A collection of named fields of differing data types. <br>`STRUCT<name:type, ...>` | `address STRUCT<city:STRING, zip:INT>` <br>Accessed via dot notation: `address.city` |
| **`UNION`** | Holds exactly one value from a predefined set of types. (Rarely used). | `val UNIONTYPE<int, double, string>` |

## 3. Partitioning vs. Bucketing

Partitioning and Bucketing are physical layout optimizations designed to drastically reduce the amount of HDFS data read during queries (avoiding full-table scans).

| Feature | Partitioning | Bucketing |
| :--- | :--- | :--- |
| **Mechanism** | Splits a table logically based on a key column value. | Splits a table physically based on the hash value of a column. |
| **Storage Structure** | Creates **subdirectories** on HDFS. | Creates **fixed-count files** inside the directory. |
| **Key Column** | Must use a **virtual column** not present in the raw data file. | Must use an **existing column** that is part of the table schema. |
| **Cardinality** | Best for columns with **low cardinality** (e.g., State, Year, Country). | Best for columns with **high cardinality** (e.g., UserID, TransactionID). |
| **File Sizing** | Can lead to "small files problem" if over-partitioned. | Restricts file counts to a set number of buckets, preventing small files. |

### HDFS Directory Layout Comparison
* **Partitioned Table** (`PARTITIONED BY (country STRING)`):
  ```text
  /user/hive/warehouse/sales_db.db/sales/
  ├── country=US/
  │   └── data_file.txt
  ├── country=IN/
  │   └── data_file.txt
  ```
* **Bucketed Table** (`CLUSTERED BY (user_id) INTO 4 BUCKETS`):
  ```text
  /user/hive/warehouse/sales_db.db/users/
  ├── 000000_0  (Contains user_ids hashing to bucket 0)
  ├── 000001_0  (Contains user_ids hashing to bucket 1)
  ├── 000002_0  (Contains user_ids hashing to bucket 2)
  └── 000003_0  (Contains user_ids hashing to bucket 3)
  ```

## 4. Managed (Internal) vs. External Tables

Hive supports two classes of table ownership:

```
Managed Table:
   DROP TABLE ---> [Metastore Schema deleted] AND [HDFS Data deleted]

External Table:
   DROP TABLE ---> [Metastore Schema deleted] ONLY. (Raw HDFS data preserved!)
```

### 1. Managed Tables (Internal Tables)
* **Definition**: Hive owns both the metadata and the raw data blocks.
* **Storage Location**: Typically stored inside Hive's default warehouse path (`/user/hive/warehouse/`).
* **Drop Behavior**: Dropping the table via `DROP TABLE` deletes the catalog schema inside the Metastore **AND** physically purges the data files off the HDFS filesystem.
* **Use Case**: Best for temporary tables, intermediate ETL processing, or datasets wholly managed within Hive.

### 2. External Tables
* **Definition**: Hive owns only the table definition (metadata). The raw data is managed outside of Hive (e.g., loaded from a separate ETL pipeline).
* **Storage Location**: Can point to any location in HDFS using the `LOCATION` keyword.
* **Drop Behavior**: Dropping the table via `DROP TABLE` deletes **only** the schema description from the Metastore database. The files stored in HDFS remain completely intact.
* **Use Case**: Best for sharing raw files with other tools (Spark, Pig, MapReduce) where Hive is only used for querying, or when raw data must not be accidentally deleted.

## 5. DDL/DML HiveQL Operations (Code Examples)

### A. Creating Managed vs. External Tables
```sql
-- Creating a Managed Table
CREATE TABLE IF NOT EXISTS employee_managed (
    emp_id INT,
    emp_name STRING,
    salary FLOAT
)
ROW FORMAT DELIMITED
FIELDS TERMINATED BY ','
STORED AS TEXTFILE;

-- Creating an External Table pointing to a custom location
CREATE EXTERNAL TABLE IF NOT EXISTS employee_external (
    emp_id INT,
    emp_name STRING,
    salary FLOAT
)
ROW FORMAT DELIMITED
FIELDS TERMINATED BY ','
LOCATION '/data/external/employees/';
```

### B. Creating and Loading Partitioned Tables

#### 1. Static Partitioning
In static partitioning, the developer manually specifies the partition value during the write step.

```sql
-- Create Partitioned Table
CREATE TABLE IF NOT EXISTS customer_partitioned (
    cust_id INT,
    cust_name STRING
)
PARTITIONED BY (country STRING, state STRING)
ROW FORMAT DELIMITED
FIELDS TERMINATED BY ',';

-- Loading Data into static partition
LOAD DATA INPATH '/tmp/us_ca_customers.csv' 
INTO TABLE customer_partitioned 
PARTITION (country='US', state='CA');
```

#### 2. Dynamic Partitioning
In dynamic partitioning, Hive automatically scans the last columns in the select statement to determine and create partition folders dynamically.

```sql
-- First, enable Dynamic Partitioning settings in the session
SET hive.exec.dynamic.partition = true;
SET hive.exec.dynamic.partition.mode = nonstrict;

-- Insert into partitioned table dynamically from a staging table
INSERT INTO TABLE customer_partitioned PARTITION(country, state)
SELECT cust_id, cust_name, country_col, state_col 
FROM staging_customer_table;
```

### C. Creating Bucketed Tables
```sql
-- Create Bucketed Table
CREATE TABLE IF NOT EXISTS users_bucketed (
    user_id INT,
    username STRING,
    email STRING
)
CLUSTERED BY (user_id) INTO 4 BUCKETS
ROW FORMAT DELIMITED
FIELDS TERMINATED BY ',';

-- Loading data into a bucketed table requires setting enforce bucketing
-- Note: In Hive 2.x+, bucketing is enforced automatically by default
SET hive.enforce.bucketing = true;

INSERT OVERWRITE TABLE users_bucketed
SELECT user_id, username, email FROM staging_users;
```

## 6. HiveQL Operators and Functions

HiveQL supports standard SQL operators and built-in functions:

### A. Operators
* **Relational Operators**: `=`, `<=>` (Null safe equality), `<>`, `!=`, `<`, `>`, `<=`, `>=`, `LIKE`, `RLIKE` (Java Regex support).
* **Arithmetic Operators**: `+`, `-`, `*`, `/`, `%`, `&` (Bitwise AND), `|` (Bitwise OR).
* **Logical Operators**: `AND`, `OR`, `NOT`.

### B. Built-In Functions (Examples)
* **String Operations**: `concat(str1, str2)`, `substr(str, start, len)`, `lower(str)`, `upper(str)`.
* **Date Operations**: `year(date)`, `month(date)`, `datediff(date1, date2)`, `date_add(date, days)`.
* **Aggregations**: `count()`, `sum()`, `avg()`, `min()`, `max()`.

### C. Analytical / Windowing Functions
Essential for complex analytical queries:
```sql
-- Calculate employee rank based on salary within departments
SELECT 
    emp_name, 
    department, 
    salary,
    RANK() OVER (PARTITION BY department ORDER BY salary DESC) as rank,
    DENSE_RANK() OVER (PARTITION BY department ORDER BY salary DESC) as dense_rank,
    ROW_NUMBER() OVER (PARTITION BY department ORDER BY salary DESC) as row_num
FROM employee_managed;
```

### D. Writing User-Defined Functions (UDFs) in Java
If a required function does not exist in Hive, developers can compile custom Java code:

```java
package bigdata.hive.udf;

import org.apache.hadoop.hive.ql.exec.UDF;
import org.apache.hadoop.io.Text;

// A custom UDF that appends a suffix to a string
public class AppendSuffixUDF extends UDF {
    public Text evaluate(Text input, String suffix) {
        if (input == null) return null;
        return new Text(input.toString() + suffix);
    }
}
```

How to use the custom UDF in Hive:
```sql
-- 1. Add your compiled jar to Hive
ADD JAR /path/to/my-hive-udfs.jar;

-- 2. Create a temporary function pointing to the class
CREATE TEMPORARY FUNCTION append_suffix AS 'bigdata.hive.udf.AppendSuffixUDF';

-- 3. Run the function in HiveQL
SELECT append_suffix(emp_name, '_Associate') FROM employee_managed;
```
