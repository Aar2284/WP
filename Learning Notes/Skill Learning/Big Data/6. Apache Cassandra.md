## 1. Cassandra Architecture

**Apache Cassandra** is a highly scalable, distributed, wide-column NoSQL database designed to handle large volumes of structured and semi-structured data across multiple commodity servers with no single point of failure. It was originally developed at Facebook and modeled after Amazon's Dynamo (for distributed systems architecture) and Google's Bigtable (for tabular storage engine).

### Peer-to-Peer Ring Topology
Unlike master-slave architectures (Hadoop/HBase), Cassandra uses a **decentralized, peer-to-peer masterless system**. Every node in the ring is identical and can process read or write requests.

```
                  Node 1 (Tokens 0 - 25B)
                 /                       \
                /                         \
  Node 4 (75B - 100B)                  Node 2 (25B - 50B)
                \                         /
                 \                       /
                     Node 3 (50B - 75B)
```

1. **Gossip Protocol**: 
   * A decentralized communication mechanism. Nodes periodically (every 1 second) swap information with up to 3 random nodes about themselves and all other nodes they know.
   * Helps determine node health, token ranges, and cluster membership without a centralized master directory.
2. **Snitches**: 
   * Maps nodes to physical racks and datacenters. Ensures Cassandra understands network topology to distribute replicas across different physical hardware.

### Replication and Consistency Levels

#### 1. Replication Factor (RF)
The total number of nodes in the cluster that will store a copy of the same row.
* If `RF = 3`, three separate nodes in the cluster hold copies of every record.

#### 2. Consistency Levels (CL)
Specifies how many replica nodes must acknowledge a read or write operation before returning success to the client.

| Consistency Level | Definition |
| :--- | :--- |
| **`ONE`** | Only 1 replica node must respond. (Fastest, low latency, but risk of stale reads). |
| **`QUORUM`** | A majority of replica nodes must respond. <br>Formula: `(RF / 2) + 1` (integer division). E.g., for `RF = 3`, Quorum is `2`. |
| **`LOCAL_QUORUM`** | A majority of replicas in the local datacenter must respond. (Optimizes multi-DC deployments). |
| **`ALL`** | All replica nodes must respond. (Strongest consistency, but high write latency; fails if any replica is down). |

#### 3. Strong Consistency Formula (Tunable Consistency)
Cassandra guarantees **Strong Consistency** (reads always return the most up-to-date write) if the read and write consistency levels satisfy:

$$\text{Read CL} + \text{Write CL} > \text{Replication Factor (RF)}$$

* **Example**: If `RF = 3` and you configure `Write CL = QUORUM` (2 nodes) and `Read CL = QUORUM` (2 nodes):
  $$2 + 2 = 4 > 3$$
  This guarantees strong consistency because at least one node is part of both the write quorum and the read quorum, ensuring the client reads the latest write.

### Detailed Write and Read Paths

#### The Write Path (Highly Optimized)
Cassandra's write path is completely sequential, making it extremely fast for write-heavy applications.

```
[Write Request] ---> [CommitLog (Disk - sequential append)]
                ---> [MemTable (RAM - sorted structure)]
                          | (When MemTable is full)
                          v
                     [SSTables (Disk - sorted, immutable files)]
```

1. **Coordinator Receipt**: Any node (acting as the Coordinator) receives the write and hashes the key.
2. **CommitLog Append**: The write is sequentially appended to the node's **CommitLog** on disk for crash recovery.
3. **MemTable Write**: The write is cached in an in-memory sorted buffer called the **MemTable**.
4. **Write Acknowledgment**: Once written to both CommitLog and MemTable, the write succeeds.
5. **Flush to SSTable**: When the MemTable is full (or the commit log segment is filled), it is flushed to disk as an immutable **SSTable (Sorted String Table)**. 
   * Because SSTables are immutable, Cassandra **never updates data in-place on disk**, avoiding expensive random disk seeks.

#### The Read Path
Reading is more complex than writing since the latest version of a record might be spread across multiple SSTables on disk and the MemTable.

1. **Check MemTable**: Look for the key in the in-memory MemTable.
2. **Bloom Filter Evaluation**: For disk files, read the **Bloom Filter** (stored in RAM) for each SSTable. The Bloom filter determines with 100% accuracy if a key does *not* exist in an SSTable (avoiding unnecessary disk reads).
3. **Key Cache Check**: Checks the Key Cache to locate the in-memory offset of the partition key within the SSTable.
4. **Row Index / Partition Summary**: If cache misses, scans the Partition Summary and Partition Index on disk to locate the data.
5. **Merge Data**: Read the data from all target SSTables and the MemTable, resolve conflicts by comparing timestamps, and return the latest version.

## 2. Cassandra Data Model

### Data Hierarchy
* **Keyspace**: The outermost container (analogous to a schema or database). Defines replication strategy and factor.
* **Table (Column Family)**: A logical schema containing rows and columns.
* **Row**: Identified by a unique primary key. Contains columns.

### Primary Key Architecture
The primary key structure determines data distribution and on-disk sorting:

```sql
PRIMARY KEY (partition_key_col, clustering_col_1, clustering_col_2)
```

1. **Partition Key**:
   * The first column defined inside the primary key.
   * **Role**: Cassandra hashes the value of this column using the Murmur3Partitioner to generate a token, which determines which node in the ring will store that partition.
   * All rows with the same Partition Key are stored physically on the same node.
2. **Clustering Columns**:
   * Any subsequent columns declared in the primary key.
   * **Role**: Determines the **on-disk sorting order** of columns within a single partition.
   * Enables rapid range queries within a specific partition (e.g., retrieving log times sorted chronologically).

### Wide Rows vs. Narrow Rows
* **Narrow Rows**: A partition contains exactly one row (only a Partition Key is defined, no clustering columns).
* **Wide Rows**: A single partition key contains millions of sorted rows mapped underneath it (using clustering columns). E.g., partition key is `sensor_id`, clustering key is `timestamp`.

## 3. CQL (Cassandra Query Language) DDL & DML Commands

CQL is similar to SQL but lacks support for joins, nested subqueries, and arbitrary group-by aggregations.

### A. Core CQL Data Types
* **Numeric**: `INT`, `BIGINT`, `FLOAT`, `DOUBLE`, `DECIMAL`, `VARINT`.
* **String**: `TEXT`, `VARCHAR`, `ASCII`.
* **Identifier**: `UUID` (Universally Unique ID), `TIMEUUID` (Type 1 UUID containing a timestamp, used for conflict-free chronological sorting).
* **Date/Time**: `TIMESTAMP`, `DATE`, `TIME`.
* **Collections**: `LIST<type>` (ordered list), `SET<type>` (unique values), `MAP<type, type>` (key-value pairs).

### B. Keyspace & Table Operations (DDL)
```sql
-- 1. Create a Keyspace with Simple Strategy (single datacenter)
CREATE KEYSPACE IF NOT EXISTS academy_vault 
WITH replication = {
    'class': 'SimpleStrategy', 
    'replication_factor': 3
};

-- Switch to the keyspace
USE academy_vault;

-- 2. Create a Table with Clustering Columns
CREATE TABLE IF NOT EXISTS student_activity (
    student_id UUID,
    activity_date DATE,
    activity_id TIMEUUID,
    activity_name TEXT,
    duration_mins INT,
    tags SET<TEXT>,
    PRIMARY KEY (student_id, activity_date, activity_id)
) WITH CLUSTERING ORDER BY (activity_date DESC);

-- Partition Key: student_id
-- Clustering Columns: activity_date and activity_id (sorted descending by date)
```

### C. Data Operations (DML)
```sql
-- 1. Insert Data
INSERT INTO student_activity (student_id, activity_date, activity_id, activity_name, duration_mins, tags)
VALUES (
    e68a3568-d0df-11ee-876a-acde48001122, 
    '2026-05-27', 
    now(), -- Generates a TIMEUUID
    'Big Data Lecture', 
    90, 
    {'hadoop', 'cassandra'}
);

-- 2. Select / Read Data
SELECT * FROM student_activity 
WHERE student_id = e68a3568-d0df-11ee-876a-acde48001122;

-- 3. Update Data (UPSERT behavior)
-- Note: In Cassandra, update and insert are structurally identical. If the key exists, it overwrites; else, it inserts.
UPDATE student_activity 
SET duration_mins = 100 
WHERE student_id = e68a3568-d0df-11ee-876a-acde48001122 
  AND activity_date = '2026-05-27' 
  AND activity_id = 1c402120-d0e0-11ee-876a-acde48001122;

-- 4. Delete Data (Creates a Tombstone marker)
DELETE duration_mins FROM student_activity 
WHERE student_id = e68a3568-d0df-11ee-876a-acde48001122 
  AND activity_date = '2026-05-27' 
  AND activity_id = 1c402120-d0e0-11ee-876a-acde48001122;
```

## 4. Advanced CQL: Filtering, Indexing, and `ALLOW FILTERING`

### A. Query Filtering Restrictions
Because Cassandra partitions data across different nodes based on the partition key, **all queries must specify the partition key** in the `WHERE` clause. 
* If you query a non-partition column:
  ```sql
  -- Invalid Query (Throws Error)
  SELECT * FROM student_activity WHERE activity_name = 'Big Data Lecture';
  ```
  Cassandra blocks this query because it would require scanning every single node in the cluster (a full-cluster scan), defeating the purpose of distributed keys.

### B. The `ALLOW FILTERING` Clausule
If you append `ALLOW FILTERING`, you force Cassandra to bypass the warning and perform the scan anyway:
```sql
-- Dangerous Query (Resource intensive)
SELECT * FROM student_activity 
WHERE activity_name = 'Big Data Lecture' 
ALLOW FILTERING;
```
> [!WARNING]
> Avoid using `ALLOW FILTERING` in production. It consumes high CPU and network resources across all nodes and can degrade cluster performance. It should only be used for testing small datasets.

### C. Secondary Indexing
To query non-partition columns efficiently without scanning the cluster, you can create a **Secondary Index**:
```sql
-- Create Secondary Index on activity_name
CREATE INDEX IF NOT EXISTS activity_name_idx 
ON student_activity (activity_name);

-- Now this query compiles and runs efficiently without ALLOW FILTERING
SELECT * FROM student_activity WHERE activity_name = 'Big Data Lecture';
```
* **Limitation**: Secondary indexes are local indexes stored on each node. They are highly efficient for low-cardinality values (e.g., Status = 'Active') but scale poorly for high-cardinality values (e.g., Email addresses).

## 5. Cassandra Administration & Compaction

### The Need for Compaction
Because SSTables are immutable:
* **Updates** write a new version of the row with a newer timestamp in a new SSTable file.
* **Deletes** do not delete data in-place; they write a delete marker called a **Tombstone** containing the deletion timestamp.
* Over time, a single row key will have values spread across dozens of SSTables, along with obsolete tombstones, causing **Read Amplification** (degrading read performance since multiple files must be searched and merged).

### What is Compaction?
**Compaction** is a background maintenance process that merges multiple SSTables into a single new SSTable, resolving duplicate records by keeping only the value with the newest timestamp and physically removing tombstoned records.

```
SSTable 1: { key1: "v1" (ts: 10) }
SSTable 2: { key1: "v2" (ts: 20), key2: Tombstone (ts: 21) }
  \                                 /
   \                               /
    v                             v
           [COMPACTION PROCESS]
                    |
                    v
SSTable 3: { key1: "v2" (ts: 20) }  (obsolete v1 and key2 deleted!)
```

### Compaction Strategies
Cassandra supports three primary compaction strategies, configured per table based on application access patterns:

1. **Size-Tiered Compaction Strategy (STCS)**
   * **Mechanism**: Merges SSTables of similar sizes when a threshold count of files is met (default: 4 files).
   * **Pros**: Very fast, low overhead during write execution.
   * **Cons**: Consumes high disk space overhead during compaction (can require up to 50% free disk space). Can lead to stale data staying on disk for a long time.
   * **Use Case**: **Write-heavy workloads** where reads are infrequent or done mainly by primary key.

2. **Leveled Compaction Strategy (LCS)**
   * **Mechanism**: Organizes SSTables into tiers (L1, L2, L3, etc.). Each tier has a capacity limit (e.g., L1 = 10MB, L2 = 100MB), and keys inside a tier do not overlap. If L1 fills up, overlapping files are merged down to L2.
   * **Pros**: Guarantees fast, low-latency reads because 90% of reads can be fulfilled by querying a single SSTable.
   * **Cons**: High write amplification (system repeatedly merges files down tiers, consuming CPU and disk I/O).
   * **Use Case**: **Read-heavy or Mixed workloads** where reads are frequent and latency is critical.

3. **Time-Window Compaction Strategy (TWCS)**
   * **Mechanism**: Groups SSTables into time windows based on the age of the data (e.g., hourly windows). Compaction only merges SSTables within the same time window.
   * **Pros**: Extremely low overhead for time-series data.
   * **Cons**: Not suitable if data is updated out-of-order frequently.
   * **Use Case**: **Time-series datasets** or datasets using TTL (e.g., logs, metrics, sensor streams).
