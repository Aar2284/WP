## 1. HBase Fundamentals and Data Model

**Apache HBase** is an open-source, distributed, versioned, non-relational, column-oriented NoSQL database modeled after Google's Bigtable. It is designed to provide random, real-time read/write access to billions of rows and millions of columns running on top of HDFS.

### Core Data Model Concepts

HBase stores data as a multidimensional, sorted, sparse map. 

```HTML
<table>
  <thead>
    <tr>
      <th rowspan="2">ROW KEY (sorted)</th>
      <th colspan="2">personal_data</th>
      <th>professional</th>
    </tr>
    <tr>
      <th>first_name</th>
      <th>last_name</th>
      <th>role</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>"user_101"</td>
      <td>Cell(v1: "Amit")</td>
      <td>Cell(v1: "Shah")</td>
      <td>Cell(v1: "Engineer")<br>(v2: "Lead Eng")</td>
    </tr>
    <tr>
      <td>"user_102"</td>
      <td>Cell(v1: "Ria")</td>
      <td>Cell(v1: "Sen")</td>
      <td>[No Cell - Sparse]</td>
    </tr>
  </tbody>
</table>
```

1. **Table**: Composed of multiple rows and columns.
2. **Row Key**: The primary key for HBase tables. 
   * It is a byte array and is **always lexicographically sorted**.
   * Selecting an optimal row key is critical since it is the primary index for all read/write routing.
3. **Column Families**: Logical groupings of columns.
   * **Must be defined upfront** when the table is created.
   * All columns within a column family are stored together in the same physical file (StoreFile).
   * Column family names must be printable strings (e.g., `personal_data`, `professional_data`).
4. **Column Qualifiers (Columns)**: Sub-divisions of column families.
   * They are **highly dynamic** and do not need to be predefined. You can add new column qualifiers (e.g., `cf:email`, `cf:phone`) on-the-fly.
   * Accessed using the format: `ColumnFamily:ColumnQualifier` (e.g., `personal_data:first_name`).
5. **Cell**: An intersection of a Row Key, Column Family, and Column Qualifier.
   * Stored as a raw byte array.
   * **Sparse Database**: If a row does not have a value for a specific column, it consumes absolutely zero storage space (unlike relational databases that store null values).
6. **Timestamp / Versioning**: Cells can hold multiple versions of a value, distinguished by a unique 64-bit integer timestamp (defaulting to system write time).

## 2. HBase Architecture

HBase operates under a Master-Slave topology coordinated by Apache ZooKeeper.

```
                             [ZooKeeper Quorum]
                            /                  \ (Tracks active nodes)
                           v                    v
                  =================      =================
                  | Active HMaster| ---> | Backup HMaster| (Hot Standby)
                  =================      =================
                     /           \
                    v             v (DDL & Load Balancing)
  =========================      =========================
  |   RegionServer (RS1)  |      |   RegionServer (RS2)  |
  |  +-----------------+  |      |  +-----------------+  |
  |  |     Region      |  |      |  |     Region      |  |
  |  |  +------------+ |  |      |  |  +------------+ |  |
  |  |  |  MemStore  | |  |      |  |  |  MemStore  | |  |
  |  |  +------------+ |  |      |  |  +------------+ |  |
  |  |  +------------+ |  |      |  |  +------------+ |  |
  |  |  | StoreFile  | |  |      |  |  | StoreFile  | |  |
  |  |  |  (HFile)   | |  |      |  |  |  (HFile)   | |  |
  |  |  +------------+ |  |      |  |  +------------+ |  |
  |  +-----------------+  |      |  +-----------------+  |
  =========================      =========================
```

### Core Architecture Components

#### 1. HMaster
The master coordinator daemon.
* Manages DDL metadata operations (creating, altering, and deleting tables).
* Assigns Regions to RegionServers and handles load balancing.
* Coordinates recovery tasks when a RegionServer goes down.

#### 2. RegionServer
The worker daemon handling data read/write requests.
* **Region**: The basic unit of scalability in HBase. Tables are horizontally split by row key ranges into segments called **Regions**. A Region contains all rows in a specific range `[StartKey, EndKey)`.
* **MemStore**: The write buffer in local RAM. When a write request arrives, the RegionServer writes it to the WAL (for recovery) and caches it in the MemStore. Once the MemStore fills up (default 128MB), it is flushed to HDFS as an immutable **StoreFile (HFile)**.
* **Write-Ahead Log (WAL / HLog)**: A transactional record on HDFS. Writes are appended to the WAL before being written to MemStore. If a RegionServer crashes, the WAL is replayed to reconstruct the lost MemStore data.
* **Store / StoreFile (HFile)**: A Store corresponds to a column family in a region. Each Store contains a single MemStore and zero or more **StoreFiles** (which are physical HFiles stored on HDFS).

#### 3. Apache ZooKeeper
Provides coordination services for the cluster.
* Maintains HMaster leader election (Active/Backup HMasters).
* Tracks the bootstrap address of the cluster and maps which RegionServers are active.
* Clients bypass the HMaster for read/write data operations; they query ZooKeeper directly to find the RegionServer housing the row key they want.

## 3. General Commands in HBase Shell (CLI)

Below is an exhaustive table of HBase Shell administration and DML commands. Run `hbase shell` to open the CLI.

| Command Syntax | Description | Example Usage |
| :--- | :--- | :--- |
| `status` | Shows cluster status details. | `status` |
| `list` | Lists all tables in HBase. | `list` |
| `describe '<table_name>'` | Displays schema details and Column Families configuration. | `describe 'employees'` |
| `create '<table_name>', '<cf_1>', '<cf_2>'` | Creates a table with specified Column Families. | `create 'employees', 'personal', 'work'` |
| `put '<table_name>', '<row_key>', '<cf:qualifier>', '<value>'` | Inserts or updates a cell value at a row key. | `put 'employees', 'emp_01', 'personal:name', 'Alice'` <br>`put 'employees', 'emp_01', 'work:role', 'Dev'` |
| `get '<table_name>', '<row_key>'` | Retrieves all cell values for a specific row key. | `get 'employees', 'emp_01'` |
| `get '<table_name>', '<row_key>', {COLUMN => 'cf:col'}` | Retrieves a specific column cell value. | `get 'employees', 'emp_01', {COLUMN => 'personal:name'}` |
| `scan '<table_name>'` | Scans and prints all rows in the table. | `scan 'employees'` |
| `scan '<table_name>', {STARTROW => 'r1', ENDROW => 'r2'}` | Performs range scanning of row keys. | `scan 'employees', {STARTROW => 'emp_01', ENDROW => 'emp_03'}` |
| `disable '<table_name>'` | Disables a table (required before dropping or altering). | `disable 'employees'` |
| `enable '<table_name>'` | Enables a disabled table. | `enable 'employees'` |
| `alter '<table_name>', NAME => '<cf>', TTL => <seconds>` | Alters a column family property (e.g., sets TTL). | `alter 'employees', NAME => 'personal', TTL => 86400` |
| `drop '<table_name>'` | Deletes the table from HBase. | `drop 'employees'` |
| `exists '<table_name>'` | Checks if a table exists. | `exists 'employees'` |

## 4. Interacting with HBase using Java API

To interact with HBase from Java applications, we use the standard HBase client library (`hbase-client`). Below is a complete, ready-to-run Java program illustrating connections, DDL (create), and DML (put, get, scan) operations.

```java
package bigdata.hbase;

import java.io.IOException;
import org.apache.hadoop.conf.Configuration;
import org.apache.hadoop.hbase.HBaseConfiguration;
import org.apache.hadoop.hbase.TableName;
import org.apache.hadoop.hbase.client.*;
import org.apache.hadoop.hbase.util.Bytes;

public class HBaseJavaDemo {

    public static void main(String[] args) {
        // Create HBase configuration (looks for hbase-site.xml in classpath)
        Configuration config = HBaseConfiguration.create();
        config.set("hbase.zookeeper.quorum", "localhost");
        config.set("hbase.zookeeper.property.clientPort", "2181");

        // Try-with-resources automatically closes Connection object
        try (Connection connection = ConnectionFactory.createConnection(config)) {
            
            TableName tableName = TableName.valueOf("student_records");
            String columnFamily = "info";

            // 1. CREATE TABLE
            try (Admin admin = connection.getAdmin()) {
                if (!admin.tableExists(tableName)) {
                    System.out.println("Creating table 'student_records'...");
                    
                    TableDescriptor tableDescriptor = TableDescriptorBuilder.newBuilder(tableName)
                        .setColumnFamily(ColumnFamilyDescriptorBuilder.newBuilder(Bytes.toBytes(columnFamily)).build())
                        .build();
                        
                    admin.createTable(tableDescriptor);
                    System.out.println("Table created successfully.");
                } else {
                    System.out.println("Table already exists.");
                }
            }

            // 2. PUT (Insert Data)
            try (Table table = connection.getTable(tableName)) {
                System.out.println("Inserting data...");
                
                // Create a Put operation for Row Key "std_101"
                Put put = new Put(Bytes.toBytes("std_101"));
                put.addColumn(Bytes.toBytes(columnFamily), Bytes.toBytes("name"), Bytes.toBytes("John Doe"));
                put.addColumn(Bytes.toBytes(columnFamily), Bytes.toBytes("grade"), Bytes.toBytes("A"));
                put.addColumn(Bytes.toBytes(columnFamily), Bytes.toBytes("city"), Bytes.toBytes("New York"));
                
                table.put(put);
                System.out.println("Row 'std_101' inserted successfully.");
            }

            // 3. GET (Retrieve Data)
            try (Table table = connection.getTable(tableName)) {
                System.out.println("Retrieving data for row 'std_101'...");
                
                Get get = new Get(Bytes.toBytes("std_101"));
                Result result = table.get(get);
                
                byte[] nameVal = result.getValue(Bytes.toBytes(columnFamily), Bytes.toBytes("name"));
                byte[] gradeVal = result.getValue(Bytes.toBytes(columnFamily), Bytes.toBytes("grade"));
                
                System.out.println("Name: " + Bytes.toString(nameVal));
                System.out.println("Grade: " + Bytes.toString(gradeVal));
            }

            // 4. SCAN (Scan Table)
            try (Table table = connection.getTable(tableName)) {
                System.out.println("Scanning entire table...");
                
                Scan scan = new Scan();
                try (ResultScanner scanner = table.getScanner(scan)) {
                    for (Result result : scanner) {
                        byte[] rowKey = result.getRow();
                        byte[] nameVal = result.getValue(Bytes.toBytes(columnFamily), Bytes.toBytes("name"));
                        System.out.println("RowKey: " + Bytes.toString(rowKey) + 
                                           " | Name: " + Bytes.toString(nameVal));
                    }
                }
            }

        } catch (IOException e) {
            e.printStackTrace();
        }
    }
}
```

## 5. Filtering in HBase using Java API

HBase handles server-side filtering to restrict the rows returned over the network to the client. This is significantly more efficient than downloading all records and filtering them client-side.

### A. PrefixFilter
Retrieves only rows whose row keys start with a specific string prefix.

```java
package bigdata.hbase;

import org.apache.hadoop.hbase.client.*;
import org.apache.hadoop.hbase.filter.PrefixFilter;
import org.apache.hadoop.hbase.util.Bytes;
import java.io.IOException;

public class HBasePrefixFilterDemo {
    public static void scanWithPrefix(Connection connection, String tableNameStr, String prefix) throws IOException {
        Table table = connection.getTable(TableName.valueOf(tableNameStr));
        
        Scan scan = new Scan();
        // Create Prefix Filter
        PrefixFilter prefixFilter = new PrefixFilter(Bytes.toBytes(prefix));
        scan.setFilter(prefixFilter);
        
        try (ResultScanner scanner = table.getScanner(scan)) {
            for (Result res : scanner) {
                System.out.println("Found RowKey: " + Bytes.toString(res.getRow()));
            }
        }
    }
}
```

### B. SingleColumnValueFilter
Filters rows based on comparing values in a specific column qualifier (e.g., return only rows where `info:grade` equals `"A"`).

```java
package bigdata.hbase;

import org.apache.hadoop.hbase.CompareOperator;
import org.apache.hadoop.hbase.client.*;
import org.apache.hadoop.hbase.filter.SingleColumnValueFilter;
import org.apache.hadoop.hbase.util.Bytes;
import java.io.IOException;

public class HBaseColumnValueFilterDemo {
    public static void scanWithColumnFilter(Connection connection, String tableNameStr) throws IOException {
        Table table = connection.getTable(TableName.valueOf(tableNameStr));
        
        Scan scan = new Scan();
        
        // SingleColumnValueFilter arguments: ColumnFamily, ColumnQualifier, CompareOperator, Value
        SingleColumnValueFilter filter = new SingleColumnValueFilter(
            Bytes.toBytes("info"),
            Bytes.toBytes("grade"),
            CompareOperator.EQUAL,
            Bytes.toBytes("A")
        );
        
        // If true, rows that do not contain the target column family/qualifier will be skipped
        filter.setFilterIfMissing(true); 
        scan.setFilter(filter);
        
        try (ResultScanner scanner = table.getScanner(scan)) {
            for (Result res : scanner) {
                byte[] name = res.getValue(Bytes.toBytes("info"), Bytes.toBytes("name"));
                System.out.println("RowKey: " + Bytes.toString(res.getRow()) + 
                                   " | Name: " + Bytes.toString(name));
            }
        }
    }
}
```

## 6. Time To Live (TTL) for Columns in HBase

HBase supports automatic expiration of cell values using **Time To Live (TTL)** properties.

### TTL Operational Rules
* TTL is configured at the **Column Family level**, not the individual column level.
* Time values are specified in **seconds**.
* Once the current time exceeds `TimestampOfCell + TTL`, the cell is marked for expiration.
* **Deletion Lifecycle**: Expired cells are immediately hidden from query `get` or `scan` operations. However, the physical bytes remain on HDFS inside StoreFiles until the next **Major Compaction** runs, which physically purges the expired records.

### A. Setting TTL via HBase Shell CLI
```bash
# Set TTL to 1 day (86400 seconds) during Table Creation
create 'logs', {NAME => 'metadata', TTL => '86400'}

# Alter an existing table to update TTL to 1 hour (3600 seconds)
disable 'logs'
alter 'logs', NAME => 'metadata', TTL => '3600'
enable 'logs'
```

### B. Setting TTL in Java API
```java
package bigdata.hbase;

import org.apache.hadoop.hbase.TableName;
import org.apache.hadoop.hbase.client.*;
import org.apache.hadoop.hbase.util.Bytes;
import java.io.IOException;

public class HBaseTtlDemo {
    public static void createTableWithTtl(Connection connection) throws IOException {
        Admin admin = connection.getAdmin();
        TableName tableName = TableName.valueOf("temporary_logs");
        
        // Configure Column Family with a TTL of 1 hour (3600 seconds)
        ColumnFamilyDescriptor cfDescriptor = ColumnFamilyDescriptorBuilder.newBuilder(Bytes.toBytes("log_cf"))
            .setTimeToLive(3600) // TTL in seconds
            .build();
            
        TableDescriptor tableDescriptor = TableDescriptorBuilder.newBuilder(tableName)
            .setColumnFamily(cfDescriptor)
            .build();
            
        admin.createTable(tableDescriptor);
        System.out.println("Table with TTL created successfully.");
        admin.close();
    }
}
```
