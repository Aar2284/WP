## Bucketing vs Partitioning in Hive: When to Use What

Partitioning and bucketing are two powerful optimization techniques in Hive that organize data for faster queries. However, they serve different purposes and are suitable for different scenarios. Choosing the wrong strategy can lead to performance nightmares—too many partitions can crash your Metastore, while improper bucketing can make joins slower than full table scans. This guide helps you make the right choice.

---

## Quick Decision Matrix

| **Scenario** | **Use Partitioning** | **Use Bucketing** | **Use Both** |
|--------------|---------------------|-------------------|--------------|
| Filtering by a specific column (e.g., date, country) | ✅ Primary choice | ❌ | ✅ Partition on filter, bucket on join key |
| Frequent JOIN operations on a column | ❌ | ✅ Primary choice | ✅ Bucket on join key |
| Need data sampling (random representative subset) | ❌ | ✅ | - |
| Column has very high cardinality (e.g., user_id, transaction_id) | ❌ (risks partition explosion) | ✅ | - |
| Column has low cardinality (e.g., region, status) | ✅ | - | - |
| Want to reduce full table scans | ✅ | ❌ | - |
| Want to improve GROUP BY performance on a column | ❌ | ✅ | - |
| Data size is small (< 1 GB) | ❌ (overhead outweighs benefit) | ❌ | ❌ |

## Understanding Partitioning

### What Partitioning Does

Partitioning physically separates data into subdirectories based on partition column values. Each distinct partition value gets its own directory.

**Example directory structure:**
```
/user/hive/warehouse/sales.db/transactions/
├── year=2024/
│   ├── month=01/
│   │   ├── part-0001
│   │   └── part-0002
│   └── month=02/
│       └── part-0001
└── year=2025/
    └── month=01/
        └── part-0001
```

### When to Use Partitioning

#### ✅ Good Use Cases for Partitioning

**1. Date/Time Based Filtering (Most Common)**
```sql
-- Query only scans the relevant partition directory
SELECT * FROM transactions 
WHERE year = 2024 AND month = 1;
-- Only reads /year=2024/month=01/ - NOT the entire table
```

**2. Geographic Filtering**
```sql
-- When users frequently query by country/region
CREATE TABLE user_events (
    user_id STRING,
    event_type STRING,
    event_data STRING
)
PARTITIONED BY (country STRING, city STRING);
```

**3. Data Lifecycle Management (Retention)**
```sql
-- Easy to drop old partitions
ALTER TABLE logs DROP PARTITION (dt='2024-01-01');

-- Old data is physically deleted from HDFS
```

**4. Loading Data Incrementally**
```sql
-- Load data for a specific date only
INSERT OVERWRITE TABLE sales PARTITION (sale_date='2024-01-15')
SELECT * FROM staging WHERE sale_date='2024-01-15';
```

**5. Regulatory Data Segregation**
```sql
-- Separate PII data by region for GDPR/CCPA compliance
PARTITIONED BY (jurisdiction STRING)  -- 'EU', 'US', 'APAC'
```

#### ❌ Bad Use Cases for Partitioning

**1. High Cardinality Columns (Partition Explosion)**
```sql
-- TERRIBLE IDEA: Partitioning on user_id (millions of values)
CREATE TABLE user_actions (
    action STRING,
    timestamp BIGINT
)
PARTITIONED BY (user_id STRING);  -- 10M users = 10M directories!

-- Problems:
-- - Metastore overhead (millions of partition entries)
-- - NameNode memory pressure (each directory is an object)
-- - Listing partitions becomes extremely slow
```

**2. Columns with Skewed Distribution**
```sql
-- Partitioning on status with 99% 'active', 1% 'inactive'
-- Most queries scan 99% of partitions anyway
```

**3. Small Datasets (< 1-5 GB)**
```sql
-- Overhead of partition management outweighs benefits
-- A full scan of 500MB is faster than partition metadata overhead
```

**4. Columns Always Used in Range Queries (Not Equality)**
```sql
-- Partitioning is optimized for equality, not ranges
WHERE price BETWEEN 100 AND 200;  -- Partitioning on price is inefficient
-- Better to use bucketing or just regular column
```

### Partitioning Limits and Safeguards

```sql
-- Set maximum partitions per table (prevents explosion)
SET hive.exec.max.dynamic.partitions = 1000;  -- Default
SET hive.exec.max.dynamic.partitions.pernode = 100;

-- Protect against accidental partition explosion
SET hive.exec.max.created.files = 100000;
SET hive.error.on.empty.partition = true;
```

## Understanding Bucketing

### What Bucketing Does

Bucketing organizes data within each partition (or table) into a fixed number of files using a hash function on the bucket column(s). All rows with the same bucket column value go into the same bucket file.

**Example: Bucketing by user_id into 4 buckets**
```
/user/hive/warehouse/sales.db/user_events/
├── bucket_0000 (users with hash%4 = 0)
├── bucket_0001 (users with hash%4 = 1)
├── bucket_0002 (users with hash%4 = 2)
└── bucket_0003 (users with hash%4 = 3)
```

### When to Use Bucketing

#### ✅ Good Use Cases for Bucketing

**1. Frequent JOINs on a Column (Most Important)**
```sql
-- Both tables bucketed on the join key with same number of buckets
CREATE TABLE orders (
    order_id INT,
    user_id INT,
    amount DECIMAL
)
CLUSTERED BY (user_id) INTO 256 BUCKETS;

CREATE TABLE user_profiles (
    user_id INT,
    name STRING,
    email STRING
)
CLUSTERED BY (user_id) INTO 256 BUCKETS;

-- This join becomes a Bucket MapJoin - no shuffling!
SELECT * FROM orders o JOIN user_profiles u ON o.user_id = u.user_id;
-- Mapper processing bucket 5 of orders only reads bucket 5 of user_profiles
```

**Why this is powerful**: Without bucketing, a join requires shuffling both tables across the network. With bucketing on the join key, each mapper processes matching buckets locally.

**2. Efficient Sampling**
```sql
-- Get a representative 10% of data (not just first N rows)
SELECT * FROM user_events TABLESAMPLE(BUCKET 3 OUT OF 10 ON user_id);

-- For true random sampling across entire dataset
SELECT * FROM large_table TABLESAMPLE(1 PERCENT);
-- Bucketing ensures each bucket is a representative sample
```

**3. Improving GROUP BY Performance**
```sql
-- When bucketed on the GROUP BY column, partial aggregation is more efficient
SELECT user_id, COUNT(*) FROM events 
GROUP BY user_id;
-- Data is already distributed by user_id, reducing shuffle
```

**4. Map-Side Joins for Large Tables**
```sql
-- Even when both tables are large, bucket map join works if:
-- - Both bucketed on join key
-- - Number of buckets in one table is multiple of other
SET hive.optimize.bucketmapjoin = true;

-- Enables map-side join without loading entire small table in memory
```

**5. Handling Data Skew in Joins**
```sql
-- Bucketing distributes skewed keys more evenly than partitioning
CLUSTERED BY (hot_user_id, salt) INTO 512 BUCKETS;
```

#### ❌ Bad Use Cases for Bucketing

**1. Filtering by Bucket Column Without a Join**
```sql
-- Bucketing doesn't help with WHERE filters
SELECT * FROM user_events WHERE user_id = 12345;
-- Still needs to read ALL buckets to find user_id 12345
-- Partitioning on user_id would help, but that's often a bad idea
```

**2. Very Small Datasets (< 100 MB)**
```sql
-- Overhead of multiple files (metadata, small I/O) hurts performance
-- A single file with index is often faster
```

**3. Frequently Changing Bucket Column**
```sql
-- Changing bucket column requires rewriting entire table
-- Bucketing is for stable query patterns
```

**4. Columns with Very Low Cardinality**
```sql
-- Bucketing on status ('active', 'inactive', 'pending')
-- Only 3 buckets, but data may be skewed
-- Partitioning would be better for low cardinality
```

## Partitioning vs Bucketing: Side-by-Side Comparison

| **Aspect** | **Partitioning** | **Bucketing** |
|------------|-----------------|---------------|
| **Physical organization** | Separate directories (subfolders) | Separate files within a directory |
| **Best for** | Query filtering (WHERE) | Joins, sampling, GROUP BY |
| **Column cardinality** | Low to medium (best with < 10K distinct values) | Any (works with high cardinality) |
| **Data distribution** | By value (exact match) | By hash (approximate distribution) |
| **Query benefit** | Partition pruning (skip entire directories) | Local joins (no shuffle) |
| **Number of files** | 1+ files per partition | Fixed number of files |
| **Can update easily** | Yes (add/drop partitions) | No (requires rewriting table) |
| **Metastore impact** | Each partition is a metadata entry | Only table metadata (not per bucket) |
| **Recovery after failure** | Partial (other partitions unaffected) | Full (must rebuild all buckets) |
| **Data skew handling** | Poor (skewed partitions) | Good (hash distributes evenly) |

## When to Use BOTH Partitioning and Bucketing

The real power comes from combining both strategies. Partition first (for coarse filtering), then bucket within each partition (for join optimization).

### Pattern: Partition on Date, Bucket on User ID

```sql
CREATE TABLE user_activity (
    user_id INT,
    activity_type STRING,
    activity_data STRING
)
PARTITIONED BY (dt STRING)           -- Partition for date filtering
CLUSTERED BY (user_id) INTO 256 BUCKETS;  -- Bucket for joins/sampling
```

**Directory structure:**
```
/user/hive/warehouse/db.db/user_activity/
├── dt=2024-01-01/
│   ├── bucket_0000 (users with hash%256 = 0)
│   ├── bucket_0001
│   └── ...
├── dt=2024-01-02/
│   ├── bucket_0000
│   └── ...
```

**Benefits:**
- Queries with date filter: Partition pruning reads only relevant date directories
- Joins on user_id within a date range: Bucket map join works within each partition
- Sampling: Get representative samples across any date range

### Example Query Using Both

```sql
-- Efficient: Partition pruning first, then bucket map join
SELECT a.user_id, a.activity_type, b.user_name
FROM user_activity a
JOIN user_profiles b ON a.user_id = b.user_id
WHERE a.dt BETWEEN '2024-01-01' AND '2024-01-31';  -- Partition pruning
-- Within each day's partition, bucket map join on user_id
```

## Practical Decision Flowchart

```
Start: Do you have a column frequently used in WHERE clauses?
    │
    ├─ YES → Is cardinality < 5000-10000?
    │           │
    │           ├─ YES → Partition on that column
    │           │
    │           └─ NO → Use bucketing + consider other optimization
    │
    └─ NO → Move to join check

Do you frequently JOIN on a specific column?
    │
    ├─ YES → Bucket both tables on the join column
    │         (Use same number of buckets for best results)
    │
    └─ NO → Move to sampling

Do you need to take representative samples frequently?
    │
    ├─ YES → Bucket on a column with even distribution
    │
    └─ NO → Consider if the table needs optimization at all

Table size < 1-5 GB?
    │
    ├─ YES → Skip both (full scan is fine)
    │
    └─ NO → Apply appropriate strategy
```

## Common Anti-Patterns to Avoid

### Anti-Pattern 1: Partitioning on Timestamp (Too Granular)

```sql
-- BAD: Partitioning on full timestamp (millions of partitions)
PARTITIONED BY (event_time TIMESTAMP)  -- Each second = new partition!

-- GOOD: Partition on date, bucket on time-of-day if needed
PARTITIONED BY (event_date STRING)
CLUSTERED BY (hour(event_time)) INTO 24 BUCKETS;
```

### Anti-Pattern 2: Bucketing the Wrong Column

```sql
-- BAD: Bucketing on a column you never join on
CLUSTERED BY (row_id) INTO 100 BUCKETS;  -- row_id is unique, no joins

-- GOOD: Bucket on foreign key used in frequent joins
CLUSTERED BY (customer_id) INTO 100 BUCKETS;
```

### Anti-Pattern 3: Too Many or Too Few Buckets

```sql
-- BAD: Only 2 buckets for 1TB of data
CLUSTERED BY (user_id) INTO 2 BUCKETS;  -- Huge files, no parallelism

-- BAD: 10,000 buckets for 1GB of data
CLUSTERED BY (user_id) INTO 10000 BUCKETS;  -- Many tiny files, NameNode overhead

-- GOOD: Rule of thumb - 2-4 buckets per reducer available
-- 256-1024 buckets for most production tables (100GB-10TB)
```

### Anti-Pattern 4: Partitioning on Unused Columns

```sql
-- BAD: Partitioning on columns never used in WHERE
PARTITIONED BY (import_timestamp STRING, source_system STRING)
-- Queries never filter on these, so always full table scan anyway

-- GOOD: Only partition on columns that prune significant data
```

## Real-World Examples

### Example 1: E-Commerce Orders Table (100 GB, 500M rows)

**Query patterns:**
- 80% of queries filter by `order_date`
- 50% of queries join with `users` table on `user_id`
- 20% of queries sample data for analysis

**Solution: Partition + Bucket**
```sql
CREATE TABLE orders (
    order_id BIGINT,
    user_id INT,
    amount DECIMAL(10,2),
    product_id INT,
    status STRING
)
PARTITIONED BY (order_date STRING)      -- Daily partitions
CLUSTERED BY (user_id) INTO 256 BUCKETS -- For joins with users table
STORED AS ORC;
```

**Why this works:**
- Date partitioning prunes to 1-30 days for most queries (~1-30GB scanned)
- User_id bucketing enables bucket map join with users table
- ORC format adds predicate pushdown and compression

### Example 2: Clickstream Logs (10 TB, 10B rows)

**Query patterns:**
- Always filtered by `dt` (partition)
- Often grouped by `session_id`
- Rarely joined (mostly aggregation)

**Solution: Partition only**
```sql
CREATE TABLE clickstream (
    session_id STRING,
    user_agent STRING,
    url STRING,
    referrer STRING
)
PARTITIONED BY (dt STRING, hour INT)   -- Two-level partitioning
STORED AS PARQUET;
```

**Why not bucket?**
- No frequent joins on a specific column
- GROUP BY on session_id is high cardinality (bucketing won't help much)
- Partitioning alone gives 80% of benefit

### Example 3: User Profiles Table (20 GB, 50M rows)

**Query patterns:**
- Frequent joins with many fact tables on `user_id`
- Rarely filtered by user attributes directly

**Solution: Bucket only (no partitions)**
```sql
CREATE TABLE user_profiles (
    user_id INT,
    name STRING,
    email STRING,
    city STRING,
    signup_date STRING
)
CLUSTERED BY (user_id) INTO 128 BUCKETS
STORED AS ORC;
```

**Why no partitions:**
- No natural low-cardinality filter column
- User_id is high cardinality (would cause partition explosion)
- Bucketing enables efficient joins with fact tables

## Configuration Settings Summary

| **Setting** | **Purpose** | **Recommended Value** |
|-------------|-------------|----------------------|
| `hive.exec.max.dynamic.partitions` | Limit total partitions | 1000-5000 |
| `hive.exec.max.dynamic.partitions.pernode` | Limit per mapper | 100-500 |
| `hive.optimize.bucketmapjoin` | Enable bucket map join | true |
| `hive.optimize.bucketmapjoin.sortedmerge` | Enable SMB join | true for sorted buckets |
| `hive.mapjoin.bucket.cache.size` | Cache size for bucket map join | 100 (default) |
| `hive.sampling.seed` | Seed for TABLESAMPLE | 1 (or random) |

## Quick Reference Card

| **Use Partitioning When...** | **Use Bucketing When...** |
|------------------------------|---------------------------|
| ✅ WHERE clause frequently filters on a column | ✅ Tables frequently JOIN on a column |
| ✅ Column has < 5000 distinct values | ✅ Column can be high or low cardinality |
| ✅ Need to drop old data easily | ✅ Need representative data samples |
| ✅ Loading data incrementally by date | ✅ Want map-side joins without small table |
| ✅ Regulatory data segregation | ✅ GROUP BY performance on that column |
| ❌ Column has millions of distinct values | ❌ Column is never used in JOINs |
| ❌ Queries rarely filter on the column | ❌ Dataset is very small (< 1 GB) |

## Conclusion

**The Golden Rule:** Partition for filtration, bucket for joining.

- **Partitioning** is about **skipping data** you don't need (pruning). Use it for common WHERE clauses on low-to-medium cardinality columns.
- **Bucketing** is about **organizing data** for efficient joins and sampling. Use it for join keys and when you need random samples.

**Start with partitioning** if you have a natural date or category dimension. **Add bucketing** on top when you have frequent, expensive joins. Never partition on high-cardinality columns—that's what bucketing is for.

**Simple litmus test:** Ask yourself "Do I want to skip reading data (partition) or organize data for faster joins (bucket)?" The answer tells you which to use.
